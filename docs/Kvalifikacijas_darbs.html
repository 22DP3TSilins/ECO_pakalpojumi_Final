ANNOTATION

The topic of this qualification paper is “Web Application for an Eco-Friendly Products and Services Platform”. The work resulted in the platform “ECO Pakalpojumi”, a full-stack web app where users can estimate their personal carbon footprint, buy eco-friendly products, join eco challenges and chat in a community forum. The frontend is built with Vue.js 3 (Composition API) and Vite 8, the backend uses Node.js with Express.js 4.18, data is kept in SQLite 3, and authentication relies on JWT, bcryptjs and Passport.js with Google OAuth 2.0.

The paper is organised as follows: an introduction, a problem statement with a Use Case diagram, a requirements specification (input/output data, functional and non-functional requirements with wireframe sketches), the rationale behind the chosen tools, system modelling and design (architecture, ER model, data-flow diagrams), a data-structure description covering all 15 database tables, a user guide, a usability-testing report, a conclusion and a reference list.

Klimata pārmaiņas vairs nav tāla problēma, par kuru runā tikai zinātnieki ziņojumos. Karstākas vasaras, plūdi, sausums un ledāju kušana ir jau tagad redzami arī Latvijā. Arvien vairāk cilvēku saprot, ka kaut kas ir jādara, taču īsti nezina, ar ko sākt. Lielie politiskie soļi, piemēram, Parīzes klimata nolīgums vai Eiropas Zaļais kurss, atstāj iespaidu valstu līmenī, bet ikdienā cilvēks paliek viens ar jautājumu „ko tieši es pats varu mainīt?".

Eiropas Komisijas pētījumi rāda, ka aptuveni trīs no četriem ES iedzīvotājiem klimata pārmaiņas uzskata par nopietnu problēmu, tomēr tikai katrs piektais ikdienā veic kaut ko konkrētu, lai mazinātu savu ietekmi uz vidi. Galvenais iemesls nav slinkums, bet gan informācijas sadrumstalotība: CO₂ kalkulators atrodas vienā vietnē, eko produkti – citā, padomi un raksti – trešajā, bet domubiedru kopiena – ceturtajā. Lai paveiktu visu nepieciešamo, lietotājam jāatver vairākas lietotnes un jāatceras dažādas paroles, kas ātri vien noved pie atteikšanās no idejas vispār.

Šis kvalifikācijas darbs piedāvā iespējamu risinājumu – tīmekļa lietojumprogrammu „ECO Pakalpojumi", kurā galvenie ikdienas videi draudzīgas dzīves rīki ir savesti vienuviet. Lietotājs var aprēķināt savu CO₂ pēdu, sekot ikdienas eko paradumiem, iegādāties dabai draudzīgus produktus, dalīties ar pieredzi forumā, piedalīties izaicinājumos un lasīt izglītojošus rakstus, neaizverot pārlūka cilni.

Galvenās sistēmas funkcijas:

  • CO₂ pēdas kalkulators – aprēķina lietotāja individuālo gada emisiju apjomu, ņemot vērā transporta, mājas, ēdiena un dzīvesveida paradumus.
  • Aktivitāšu izsekotājs – ļauj reģistrēt mazas, ikdienišķas eko darbības (piemēram, došanos uz darbu ar velosipēdu) un redzēt to summāro ietekmi laika gaitā.
  • Eko produktu veikals – ekoloģisku preču katalogs ar īsu aprakstu par katra produkta dzīves ciklu un ietekmi uz vidi.
  • Kopienas forums – vieta, kur uzdot jautājumus, dalīties pieredzē un atrast domubiedrus.
  • Eko izaicinājumi – uzdevumi ar dažādu grūtības pakāpi, kas motivē turpināt zaļākas izvēles ikdienā ar punktu un līmeņu palīdzību.
  • Izglītības sadaļa – raksti, video un īsi ceļveži par konkrētām vides tēmām.

Sistēma ir veidota plašai auditorijai – skolēniem, studentiem, ģimenēm un cilvēkiem, kuri tikai sāk interesēties par ilgtspēju. Lietotāja saskarne ir pieejama gan latviešu, gan angļu valodā, lai to varētu izmantot arī ārpus Latvijas.

1. UZDEVUMA NOSTĀDNE

Šajā nodaļā tiek aprakstīts, ko sistēmai „ECO Pakalpojumi" jāspēj darīt: kas to lietos, kādi būs tās galvenie elementi un kādas tiesības būs katram lietotāju veidam. Šī nodaļa ir pamats nākamajām – prasību specifikācijai un projektēšanai.

1.1. Sistēmas vispārīgais apraksts

„ECO Pakalpojumi" ir tīmekļa lietojumprogramma, kas darbojas kā vienas lapas lietotne (SPA). Tas nozīmē, ka pēc pirmās lapas ielādes pārlūks neielādē jaunas HTML lapas — saturs nomainās uzreiz, un lietošana atgādina parastu mobilo aplikāciju. Tā strādā jebkurā mūsdienu pārlūkā – datorā, planšetē vai telefonā – ja vien ir interneta savienojums.

Sistēma ir veidota no divām daļām:

  • Klienta puse (frontend) – lietotāja saskarne, kas izveidota ar Vue.js 3 ietvaru, izmantojot Composition API pieeju. Klienta puse nodrošina interaktīvu, atsaucīgu un vizuāli pievilcīgu lietotāja pieredzi.
  • Servera puse (backend) – REST API serveris, kas izveidots ar Node.js un Express.js ietvaru. Servera puse nodrošina biznesa loģiku, datu apstrādi, autentifikāciju, autorizāciju un datubāzes pārvaldību.

Šādu sadalījumu sauc par klienta–servera modeli. Klients un serveris savā starpā runā ar HTTP pieprasījumiem (GET, POST, PUT, DELETE). Tāda pieeja ļauj abas puses attīstīt atsevišķi un, ja vajag, vēlāk vienu nomainīt, nediedzot otru.

1.2. Sistēmas lietotāji un to lomas

Sistēmā ir paredzētas trīs lietotāju kategorijas. Divas no tām ir reģistrēti lietotāji (parasts lietotājs un administrators), bet trešā – nereģistrēti apmeklētāji jeb viesi.

1.2.1. Neautentificēts lietotājs (viesis)

Viesis ir cilvēks, kas atvēris vietni, bet vēl nav reģistrējies vai pieteicies. Viņa iespējas ir apzināti ierobežotas – viņš var iepazīties ar saturu, bet nevar nedz pirkt, nedz reģistrēt savas eko aktivitātes. Konkrēti viesis var:

  • Apskatīt sākumlapu ar platformas pārskatu un globālās statistikas datiem.
  • Pārlūkot ekoloģisko produktu katalogu (filtrēt, kārtot, meklēt produktus).
  • Lasīt kopienas foruma ierakstus un komentārus.
  • Piekļūt izglītojošo resursu sadaļai (raksti, video, ceļveži).
  • Mainīt platformas valodu (latviešu / angļu).
  • Pārslēgt tumšo / gaišo režīmu.
  • Reģistrēties sistēmā vai pieteikties ar esošu kontu.

1.2.2. Autentificēts lietotājs (reģistrēts lietotājs)

Reģistrēts lietotājs ir tāds, kas izveidojis kontu un pieteicies sistēmā. Viņam pieejamas visas viesa iespējas un papildus vēl šādas:

  • Izmantot oglekļa pēdas nospieduma kalkulatoru ar detalizētu CO₂ emisiju aprēķinu.
  • Reģistrēt ikdienas ekoloģiskās aktivitātes un sekot līdzi savam progresam.
  • Pievienot produktus iepirkumu grozam un veikt pasūtījumus.
  • Izmantot atlaižu kodus pasūtījumu noformēšanā.
  • Apskatīt savu pasūtījumu vēsturi.
  • Izveidot foruma ierakstus, komentēt un atzīmēt „patīk".
  • Saglabāt foruma ierakstus grāmatzīmēs.
  • Pievienoties eko izaicinājumiem un sekot līdzi savam progresam.
  • Apskatīt un rediģēt savu profilu (vārds, e-pasts, avatars, bio).
  • Sekot līdzi saviem sasniegumiem, līmeņiem un XP punktiem.
  • Saņemt un pārvaldīt paziņojumus.
  • Dzēst savu kontu.

1.2.3. Administrators

Administrators ir lietotājs ar paplašinātām tiesībām – viņš pārvalda saturu un citus lietotājus. Papildus parastā lietotāja iespējām administrators var:

  • Piekļūt administrācijas panelim ar statistikas pārskatiem.
  • Pārvaldīt lietotāju kontus – apskatīt, bloķēt/atbloķēt, mainīt lomas, dzēst.
  • Pārvaldīt produktu katalogu – izveidot, rediģēt, dzēst produktus.
  • Augšupielādēt produktu attēlus.
  • Pārvaldīt pasūtījumus – apskatīt, mainīt statusu, dzēst.
  • Moderēt forumu – dzēst neatbilstošus ierakstus.
  • Apskatīt sistēmas statistiku pa laika periodiem (šodien, nedēļa, mēnesis, gads).
  • Pārvaldīt izaicinājumus un izglītojošos materiālus.

1.3. Sistēmas funkcionālais apraksts

Sistēmas funkcionalitāti var sadalīt septiņās lielās grupās:

  1. Autentifikācija un lietotāju pārvaldība – reģistrācija ar e-pastu vai Google OAuth, pieteikšanās, paroles atjaunošana, profila rediģēšana, konta dzēšana.
  2. Oglekļa pēdas nospieduma aprēķināšana – detalizēts CO₂ emisiju kalkulators ar 4 kategorijām (transports, māja, ēdiens, dzīvesveids) un aktivitāšu izsekotājs ar statistikas vizualizāciju.
  3. E-komercija – produktu kataloga pārlūkošana ar filtrēšanu un kārtošanu, iepirkumu grozs, pasūtījumu izveide ar promo kodu atbalstu, pasūtījumu vēsture.
  4. Kopienas forums – ierakstu veidošana, komentēšana (ar ligzdotajiem komentāriem), „patīk" atzīmēšana, grāmatzīmes, skatījumu uzskaite.
  5. Eko izaicinājumi – izaicinājumu saraksta pārlūkošana, pievienošanās, progresa izsekošana, punktu un līmeņu sistēma.
  6. Izglītojošie resursi – rakstu, video un ceļvežu katalogs ar meklēšanas iespēju.
  7. Administrēšana – statistikas panelis, lietotāju, produktu, pasūtījumu pārvaldība un foruma moderēšana.

2.1.2. Izejas informācijas apraksts

Izejas dati ir tas, ko sistēma atgriež lietotājam – saskarnes ekrānos vai kā API atbildes. Tālāk apkopoti galvenie izādes datu veidi.

1. Tiks ģenerēts CO₂ pēdas aprēķina rezultāts, kas satur kopējo gada oglekļa emisiju apjomu tonnās, sadalījumu pa četrām kategorijām (transports, māja, ēdiens, dzīvesveids) procentos, sektoru diagrammas vizualizāciju un personalizētus samazināšanas ieteikumus, balstoties uz lielāko ieguldījumu kopējā pēdā.

2. Tiks attēlots lietotāja profila kopsavilkums, kas satur kopējo ietaupīto CO₂ apjomu (kg), izpildīto izaicinājumu skaitu, veikto pasūtījumu skaitu, secīgo aktīvo dienu skaitu (sēriju), pašreizējo līmeni, XP punktus un progresa joslu uz nākamo līmeni.

3. Tiks ģenerēts pasūtījuma apstiprinājums, kas satur pasūtījuma numuru, datumu, produktu sarakstu ar daudzumiem un cenām, starpsummu, piemēroto atlaidi (ja izmantots promo kods), kopējo summu, piegādes informāciju un eko ietekmes aprēķinu (stādāmie koki, kompensētais CO₂).

4. Tiks attēlots produktu kataloga saraksts, kas satur produkta attēlu, nosaukumu, kategoriju, cenu, īsu aprakstu, krājuma statusu un pogu „Pievienot grozam". Saraksts ir filtrēts pēc kategorijas, cenu diapazona un kārtots pēc lietotāja izvēles.

5. Tiks attēlots iepirkumu groza saturs, kas satur visus pievienotos produktus ar attēliem, daudzumiem, vienības cenām, starpsummu, piemēroto atlaidi un kopējo summu, kā arī aprēķinātu ekoloģisko ietekmi.

6. Tiks ģenerēta nedēļas aktivitāšu statistika, kas satur joslu diagrammu ar dienas ietekmes vērtībām (kg CO₂), sektoru diagrammu ar kategoriju sadalījumu, sasniegumu žetonus, sēriju skaitītāju un mēneša mērķa progresa apli.

7. Tiks attēlots foruma ierakstu saraksts, kas satur ierakstu virsrakstus, autorus ar avatariem, publikācijas datumus, kategoriju etiķetes, skatījumu skaitu, „patīk" skaitu un komentāru skaitu. Saraksts ir filtrēts pēc kategorijas un meklēšanas atslēgvārdiem.

8. Tiks attēlots eko izaicinājumu saraksts, kas satur izaicinājuma nosaukumu, aprakstu, grūtības pakāpi, dalībnieku skaitu, atlikušo dienu skaitu, iespējamos punktus un lietotāja individuālo progresa procentu.

9. Tiks attēlots paziņojumu saraksts, kas satur paziņojuma tipu (sveiciena, pasūtījuma, sistēmas), tekstu, datumu, ikonu, saiti uz attiecīgo sadaļu un lasīšanas statusu.

10. Tiks ģenerēts administrācijas statistikas pārskats, kas satur kopējo lietotāju skaitu, jauno lietotāju skaitu pa periodiem (šodien, nedēļa, mēnesis, gads), aktīvo pasūtījumu skaitu, kopējos ieņēmumus, tendenču indikatorus (↑↓%) un grafikus.

11. Tiks attēlota globālā sistēmas statistika, kas satur kopējo reģistrēto lietotāju skaitu, kopējo ietaupīto CO₂ apjomu (kg), aprēķināto stādīto koku skaitu, kopējo izpildīto izaicinājumu skaitu, foruma ierakstu skaitu un pārdoto produktu skaitu.

12. Tiks attēloti kļūdu paziņojumi (validācijas kļūdas, autentifikācijas kļūdas, autorizācijas kļūdas, servera kļūdas) ar paskaidrojošu tekstu un norādēm par to, kā lietotājs problēmu var novērst.

2.2. Funkcionālās prasības

Funkcionālās prasības ir konkrētas darbības, ko sistēmai ir jāspēj veikt. Tās ir sadalītas pa funkcionalitātes blokiem (autentifikācija, kalkulators, e-komercija, forums un tā tālāk), lai vēlāk projektēšanas un testēšanas posmā pie tām būtu vieglāk atgriezties.

1. Reģistrācija un autentifikācija.

1.1. Sistēmai jānodrošina jauna lietotāja reģistrācija, ievadot vārdu, e-pasta adresi un paroli.

1.2. Sistēmai jāpārbauda, vai parole satur vismaz 8 simbolus, vienu lielo burtu, vienu mazo burtu un vienu ciparu.

1.3. Sistēmai jāpārbauda, vai e-pasta adrese atbilst RFC formātam un nav jau reģistrēta sistēmā.

1.4. Sistēmai jāglabā lietotāju paroles jauktā veidā, izmantojot bcryptjs ar 10 sāls kārtām.

1.5. Sistēmai jānodrošina pieteikšanās ar e-pasta adresi un paroli.

1.6. Sistēmai jānodrošina pieteikšanās ar Google kontu, izmantojot Google OAuth 2.0 protokolu.

1.7. Sistēmai pēc veiksmīgas pieteikšanās jāģenerē JWT tokens ar 24 stundu derīguma termiņu.

1.8. Sistēmai jānodrošina pieprasījumu ierobežošana autentifikācijas galapunktiem (reģistrācija – 5 pieprasījumi/15 min, pieteikšanās – 10 pieprasījumi/15 min).

1.9. Sistēmai jānodrošina lietotāja izrakstīšanās no sistēmas un sesijas datu dzēšana.

2. Lomu balstīta piekļuves kontrole.

2.1. Sistēmai jāatbalsta divas lietotāju lomas: parastais lietotājs (user) un administrators (admin).

2.2. Sistēmai jāierobežo piekļuve administrēšanas paneļa maršrutiem tikai lietotājiem ar administratora lomu.

2.3. Sistēmai jāierobežo piekļuve personīgajiem maršrutiem (profils, grozs, pasūtījumi) tikai autentificētiem lietotājiem.

2.4. Sistēmai jāļauj veidot un dzēst tikai savus datus (foruma ierakstus, komentārus, groza saturu).

3. Lietotāja profila pārvaldība.

3.1. Sistēmai jāatļauj autentificētam lietotājam apskatīt savu profilu ar personīgo statistiku.

3.2. Sistēmai jāatļauj autentificētam lietotājam rediģēt savu vārdu, e-pastu, biogrāfiju.

3.3. Sistēmai jāatļauj autentificētam lietotājam augšupielādēt avatara attēlu (JPEG, PNG, maks. 5 MB).

3.4. Sistēmai jāatļauj autentificētam lietotājam dzēst savu kontu.

3.5. Sistēmai jāaprēķina lietotāja XP punkti pēc formulas: (aktivitātes × 10) + (izaicinājumi × 50) + (pasūtījumi × 5).

3.6. Sistēmai jāaprēķina lietotāja līmenis pēc formulas: punkti ÷ 100 + 1.

4. Oglekļa pēdas nospieduma kalkulators.

4.1. Sistēmai jānodrošina CO₂ kalkulators ar četrām kategorijām: transports, māja, ēdiens, dzīvesveids.

4.2. Sistēmai jāatļauj lietotājam ievadīt transporta datus: auto km/nedēļā, auto degvielas tips, lidojumi/gadā, sabiedriskais transports stundas/nedēļā.

4.3. Sistēmai jāatļauj lietotājam ievadīt mājas patēriņa datus: elektroenerģija kWh/mēnesī, apkure m³/mēnesī, ūdens L/dienā.

4.4. Sistēmai jāatļauj lietotājam ievadīt ēdiena patēriņa datus: gaļas maltītes/nedēļā, piena produkti/dienā, atkritumu procentuālais īpatsvars.

4.5. Sistēmai jāatļauj lietotājam ievadīt dzīvesveida datus: apģērba gabali/mēnesī, sūtījumi/mēnesī, pārstrādes paradumi.

4.6. Sistēmai jāaprēķina kopējais gada CO₂ izmešu apjoms tonnās un sadalījums pa kategorijām procentos.

4.7. Sistēmai jāattēlo kalkulatora rezultāti vizuāli, izmantojot sektoru diagrammu.

4.8. Sistēmai jāģenerē personalizēti CO₂ samazināšanas ieteikumi, balstoties uz lielāko emisiju kategoriju.

5. Aktivitāšu reģistrēšana un izsekošana.

5.1. Sistēmai jānodrošina iepriekš definētu ekoloģisko aktivitāšu ātrā reģistrēšana (atjaunojamā enerģija, plastmasas izvairīšanās, koku stādīšana, sabiedriskais transports u.c.).

5.4. Sistēmai jāattēlo aktivitāšu kategoriju sadalījuma sektoru diagramma.

5.5. Sistēmai jāuzskaita un jāattēlo secīgo dienu skaits (sērija), kurās lietotājs reģistrējis vismaz vienu aktivitāti.

5.6. Sistēmai jānodrošina sasniegumu žetonu (badge) sistēma, kas tiek atbloķēta, sasniedzot noteiktus rādītājus.

5.7. Sistēmai jāattēlo apļveida progresa mērītājs uz mēneša ekoloģisko mērķi.

6. Produktu kataloga pārlūkošana.

6.1. Sistēmai jānodrošina ekoloģisko produktu katalogs ar attēliem, nosaukumiem, aprakstiem, cenām un dzīves cikla informāciju.

6.2. Sistēmai jānodrošina produktu filtrēšana pēc kategorijas (Personal Care, Accessories, Electronics, Clothing, Kitchen).

6.3. Sistēmai jānodrošina produktu filtrēšana pēc cenu diapazona un krājuma pieejamības.

6.4. Sistēmai jānodrošina produktu kārtošana pēc cenas (augoša/dilstoša) un nosaukuma (A–Z).

6.5. Sistēmai jānodrošina produktu meklēšana pēc atslēgvārdiem nosaukumā un aprakstā.

7. Iepirkumu groza pārvaldība.

7.1. Sistēmai jāļauj autentificētam lietotājam pievienot produktu grozam, norādot daudzumu.

7.2. Sistēmai jāļauj lietotājam mainīt produkta daudzumu grozā.

7.3. Sistēmai jāļauj lietotājam noņemt atsevišķu produktu no groza.

7.4. Sistēmai jāļauj lietotājam iztukšot visu grozu.

7.5. Sistēmai jāaprēķina groza starpsumma, atlaide un kopējā summa reālajā laikā.

7.6. Sistēmai jāaprēķina groza ekoloģiskā ietekme – stādāmo koku skaits un kompensētais CO₂.

8. Pasūtījumu izveidošana un apstrāde.

8.1. Sistēmai jānodrošina pasūtījuma noformēšana no iepirkumu groza, ievadot piegādes adresi un kontaktinformāciju.

8.2. Sistēmai jāatbalsta atlaižu promo kodu izmantošana: ECO10 (10%), GREEN20 (20%), PLANET15 (15%).

8.3. Sistēmai pirms pasūtījuma apstiprināšanas jāpārbauda produktu krājums.

8.4. Sistēmai pēc pasūtījuma apstiprināšanas jāsamazina produktu krājums un jāiztukšo grozs.

Trešais – ekosistēma. Vue Router, Pinia un Vue I18n nav trešo pušu pakotnes, bet pašas Vue komandas oficiāli uzturēti rīki. Tas nozīmē, ka tie savstarpēji strādā saskaņoti un nepārtraukti tiek atjaunināti vienlaikus ar pašu Vue.

Apsvēru arī alternatīvas:

  • React dara līdzīgu darbu, bet prasa vairāk papildu izvēļu (kuru maršrutētāju ņemt, kuru store, kuru formu bibliotēku). Vue tās lietas jau ir izlēmis manā vietā.
  • Angular ir spēcīgs, bet šādam projektam pārāk smags un uzspiež TypeScript, kas paildzinātu apmācību.
  • Svelte ir interesants, taču tā ekosistēma vēl ir mazāka, un kvalifikācijas darbā negribējās uzlikt likmi uz tikko nostiprinošos rīku.

Praksē noderīgākās lietas:

  • Karstā moduļu nomaiņa (HMR) – izmaiņas kodā uzreiz parādās pārlūkā, neaizverot lapas stāvokli (formās ievadītie dati paliek);
  • Optimizēts produkcijas būvējums – zem motora pārsega Vite izmanto Rollup, kas sadala kodu un izņem nelietotās daļas;
  • Tiešs Vue.js atbalsts – Vite veido tā pati komanda, kas Vue, tāpēc integrācija ir bez līkločiem;
  • Modernie standarti – jaunākā JavaScript un CSS sintakse strādā bez papildu konfigurācijas.

3.2.3. Vue Router 4.2.5 – maršrutēšana

SPA lietotnē nav klasisku lapu pārlādes – visa navigācija notiek viena dokumenta ietvaros. Lai tā darbotos cilvēkam saprotami (URL atspoguļo lapu, atpakaļ poga strādā, saiti var nosūtīt draugam), tiek izmantots Vue Router. Galvenās lietas, kam tas projektā kalpo:

  • Maršrutu definēšana – 11 maršruti (/, /login, /register, /products, /cart, /footprint, /challenges, /forum, /education, /profile, /admin) ar slinko ielādi, lai sākumā netiktu lejupielādēts viss projekts uzreiz.
  • Sargi (navigation guards) – pirms pārejas uz aizsargātu lapu (piemēram, /admin) tiek pārbaudīts, vai lietotājs ir pieteicies un vai viņam ir vajadzīgā loma.
  • Maršrutu metadati – katra maršruta meta laukos ir norādīts, vai tā piekļuvei vajadzīga autentifikācija, administratora tiesības vai arī tā ir pieejama tikai viesiem.
  • Pāreju animācijas – nelielas CSS pārejas, lai lapu maiņa neizskatās pārāk strauja.

3.2.4. Pinia 3.0.4 – stāvokļa pārvaldība

Pinia ir tā vieta, kur sistēma glabā datus, kuriem jābūt pieejamiem visās lapās – piemēram, pieteiktā lietotāja datus, JWT tokenu un lomu. Bez šādas centrālas „atmiņas" katrai komponentei būtu jāpārsūta šī informācija caur ieejas parametriem, kas ātri kļūst nepārvaldāmi.

Pinia ir Vuex pēctecis un mūsdienām piemērotāks:

  • vienkāršāks API – nav vairs atsevišķa „mutāciju" jēdziena, datus var mainīt tieši;
  • pilns Composition API atbalsts;
  • labāks TypeScript atbalsts (ja vēlāk to ievieš);
  • plakana, nesarežģīta moduļu struktūra.

3.2.5. Vue I18n 9.14.5 – internacionalizācija

Lai sistēmā darbotos valodas pārslēgšana (LV/EN), tika pievienots Vue I18n. Tulkojumi atrodas atsevišķos failos pa valodām, un komponentes tiem piekļūst caur funkciju t('key'). Kopumā ir vairāk nekā 500 tulkojumu atslēgu katrai valodai – tas aptver navigāciju, formas, kalkulatoru, forumu, izglītības sadaļu un administrēšanas paneli.

3.2.6. Axios 1.13.2 – HTTP klients

Visu API komunikāciju veic Axios. Salīdzinot ar tiešo fetch(), tas piedāvā dažas ērtības, kas šādā projektā tiešām ir vajadzīgas:

  • Pārtvērēji (interceptors) – ļauj automātiski pielikt JWT tokenu katram pieprasījumam un kļūdas apstrādāt vienā vietā;
  • Automātiska JSON parsēšana – nav jāraksta papildu .json() izsaukums;
  • Iespēja atcelt pieprasījumus, ja lietotājs ātri pārslēdz lapas;
  • Konsekventa darbība visos mūsdienu pārlūkos.

Projektā ir viens centrāls API modulis (src/utils/api.js), kurā definēta bāzes Axios instance ar bāzes URL un autorizācijas pārtvērēju. Visas komponentes lieto šo vienu instanci – ja kaut kas mainās (piemēram, vajag pievienot citu galveni), izmaiņa notiek tikai vienā vietā.

3.2.7. CSS3 ar CSS mainīgajiem – vizuālais noformējums

Vizuālajai pusei apzināti netika izvēlēts neviens gatavs CSS ietvars (Bootstrap, Tailwind un tamlīdzīgi) – tika izmantots tīrs CSS3. Galvenie iemesli:

  • Pilna kontrole pār dizainu – nav jāpielāgojas svešas bibliotēkas paradumiem;
  • CSS mainīgie – galvenās krāsas (--primary, --accent, --accent-earth) ir definētas vienā vietā; pateicoties tam, gaišo un tumšo režīmu var mainīt, pārrakstot tikai pāris vērtības;
  • CSS Grid un Flexbox – pietiek atsaucīgam izkārtojumam bez ārējām bibliotēkām;
  • Mazāks koda apjoms – nav neizmantoto klašu „balasta", ko pievienotu gatavs ietvars;
  • Iekšējās ēnu un pāreju klases dažādās intensitātēs ļauj uzturēt saskaņotu vizuālo izskatu visā sistēmā.

3.3. Backend tehnoloģiju pamatojums

3.3.1. Node.js – servera puses izpildlaika vide

Kā servera puses izpildlaika vide tika ņemts Node.js. Galvenais arguments – var uzrakstīt visu projektu vienā valodā: gan pārlūks, gan serveris, gan migrāciju skripti darbojas JavaScript valodā. Tas nozīmē, ka uzmanība nav nepārtraukti jāpārslēdz starp dažādām valodām.

  • Viena valoda visur – mazāk konteksta pārslēgšanas izstrādes laikā;
  • Asinhrons I/O – Node.js notikumu cilpa labi tiek galā ar daudziem vienlaicīgiem tīmekļa pieprasījumiem;
  • npm ekosistēma – gandrīz visam, kas vajadzīgs, jau ir gatava pakotne;
  • Liela kopiena – ja kaut kas nestrādā, atbilde uz Stack Overflow parasti jau ir.

Alternatīvas, kā Python (Django, Flask) vai PHP (Laravel), būtu noderējušas, ja projektam vajadzētu smagākus datu apstrādes algoritmus vai īpašas valodas iespējas. Šādiem samērā vienkāršiem CRUD pieprasījumiem Node.js pilnīgi pietiek.

3.3.2. Express.js 4.18.2 – backend ietvars

Express.js ir šodien populārākais Node.js tīmekļa ietvars. Tā priekšrocība ir tas, ka tas neuzspiež neko lieku – dod tikai galvenās lietas (maršrutēšanu, starpprogrammatūru, kļūdu apstrādi), bet nelej kaklā papildu struktūras vai konvencijas. Projektā Express.js strādā:

  • API galapunktu definīšanai – vairāk nekā 40 galapunkti, sagrupēti pa loģiskajām grupām (autentifikācija, produkti, grozs, forums, izaicinājumi, administrēšana);
  • Starpprogrammatūrai – JWT verifikācija, CORS, faila augšupielādes apstrāde, pieprasījumu skaita ierobežošana;
  • Kļūdu apstrādei – viena vieta, kur visas kļūdas tiek nozīmētas žurnālā un nosūtītas klientam vienotā formātā;
  • Statisko failu servēšanai – augšupielādēto attēlu piegādei no /uploads mapes.

Apsverot citus ietvarus (Fastify, Koa, Hapi), izvēli noteica kopienas izmērs: Express dokumentācija un piemēri praktiski jebkurai problēmai jau ir atrodami.

3.3.3. JSON Web Tokens (JWT) – autentifikācija

Autentifikācijai tika izvēlēta uz JWT balstīta pieeja. Tā sauktais bezstāvokļa modelis nozīmē, ka serverim nav jāuztur sesiju saraksts atmiņā – viss vajadzīgais (lietotāja ID, loma, derīguma laiks) ir iekodēts pašā tokenā. Tas ir parocīgi SPA projektiem, kur klients un serveris bieži mijiedarbojas.

  • Bez sesiju glabāšanas serverī – mērogot ir vienkāršāk;
  • Kompakts – tokens tiek nēsāts HTTP galvenē, neaizņem daudz vietas;
  • Pašpietiekams – satur lietotāja ID un lomu, tāpēc atļauju pārbaudi var izdarīt bez papildu DB vaicājuma;
  • Divu tokenu pāris – piekļuves tokens (24 h) un atjaunošanas tokens (7 dienas) – samērā labs kompromiss starp drošību un to, ka lietotājam nav nepārtraukti jāpārpiesakās.

3.3.4. bcryptjs – paroļu jaukšana

Paroles netiek glabātas atklātā veidā – pirms ierakstīšanas datubāzē tās tiek pārveidotas ar bcryptjs jaukšanas funkciju. Bibliotēka tika izvēlēta tāpēc, ka:

  • Pati ģenerē „sāli" (salt) – katrai parolei savu, tāpēc divas vienādas paroles datubāzē izskatīsies pilnīgi atšķirīgi;
  • Algoritms ir apzināti lēns – tas neietekmē parastu lietošanu, bet padara nepraktiskus uzbrukumus, kuros mēģina visas iespējamās paroles pēc kārtas;
  • Konfigurējama sarežģītība – projektā ir 10 sāls kārtas (salt rounds); šo skaitli vēlāk var paaugstināt, kad serveri kļūst ātrāki;
  • Tīrs JavaScript – nav atkarības no kompilētiem C++ moduļiem, kas vienkāršo izvietošanu.

3.3.5. Passport.js – sociālā autentifikācija

Lai dotu lietotājam iespēju pieteikties ar Google kontu (un nebūtu jāizdomā vēl viena parole), tika pievienots Passport.js ar passport-google-oauth20 stratēģiju. Passport ir gandrīz standarta risinājums šim uzdevumam Node.js pasaulē – tas atbalsta vairāk nekā 500 dažādu autentifikācijas piegādātāju, tāpēc nākotnē, ja gribētu pievienot, piemēram, Facebook vai GitHub, tas nozīmētu vien dažu rindu papildinājumu.

3.3.6. Multer – failu augšupielāde

Failu augšupielādei (galvenokārt produktu un avataru attēliem) tiek lietots Multer. Tas iekļaujas Express.js ekosistēmā kā parasta starpprogrammatūra un ietver visu vajadzīgo:

  • Faila tipa pārbaude – tiek pieņemti tikai attēlu formāti (JPEG, PNG, GIF, WebP);
  • Izmēra ierobežojums – līdz 5 MB vienam failam;
  • Glabāšana diskā – faili tiek saglabāti /uploads mapē ar unikāliem nosaukumiem, lai nepārrakstītu cits citu;
  • multipart/form-data parsēšana – Multer dara šo darbu pats, klasiskais Express to nepiedāvā.

3.4. Datu bāzes vadības sistēmas pamatojums

3.4.1. SQLite – relāciju datu bāze

Datu bāzes līmenī tika izvēlēts SQLite. Tas mazliet atšķiras no MySQL vai PostgreSQL: SQLite „datu bāze" ir vienkārši viens fails diskā, nevis atsevišķs serviss, kas jāstartē. Šim konkrētajam projektam tāda pieeja patiešām noder:

  • Bez atsevišķa servera – nav jāuztur datubāzes serviss; izstrādes, testēšanas un demonstrēšanas vidē tas ir liels atvieglojums.
  • Nulles konfigurācija – nekas nav jāiestata pirms sākšanas (nav lietotāju, tiesību, tīkla portu).
  • Pilns SQL atbalsts – SQLite saprot lielāko daļu standarta SQL, tostarp ārējās atslēgas, indeksus, JOIN un apakšvaicājumus, kas šī projekta 15+ tabulu shēmai ir nepieciešami.
  • Ātra lasīšana – tīmekļa lietotnēs lasīšanas notiek daudz vairāk nekā ierakstīšanas, un SQLite ar to lieliski tiek galā.
  • Pārnesamība – datu bāzi var pārvietot, vienkārši nokopējot vienu failu; testēšanas un nodošanas posmā tas ir liels pluss.
  • Stabilitāte – SQLite ir viena no visplašāk izmantotajām datubāzes bibliotēkām pasaulē (visi Android telefoni, pārlūki) un atbalsta visus ACID nosacījumus.

Projekta datu bāzē ir definētas šādas galvenās tabulas:

  • users – lietotāju konti ar lomām un autentifikācijas datiem;
  • products – ekoloģisko produktu katalogs;
  • cart – iepirkumu groza dati;
  • orders un order_items – pasūtījumu informācija;
  • user_activities – ekoloģisko aktivitāšu reģistrs;
  • posts, comments, post_likes, post_bookmarks, comment_likes – foruma dati;
  • challenges un user_challenges – izaicinājumu sistēma;
  • education_posts – izglītojošie materiāli;
  • notifications – paziņojumu sistēma.

6. LIETOTĀJA CEĻVEDIS

Šajā nodaļā īsi paskaidroju, kas vajadzīgs sistēmas „ECO Pakalpojumi" palaišanai, kā to uzstādīt un kā izskatās galvenie ekrāni. Mērķis ir, lai jaunu lietotāju vai pasniedzēju apjucinoss netu – lai varētu sekot līdzi ar to vien, kas šeit ir uzraksīts.

6.1. Sistēmas prasības aparatūrai un programmatūrai

„ECO Pakalpojumi" ir tīmekļa lietotne, tāpēc lielas prasības pret datoru tā neuzdod – jāstrādā jebkurā, kur darbojas modernāks pārlūks. Tomēr drošības labad tālāk uzskaitītas konkrēti ieteicamās versijas un parametri.

Operētājsistēmas, kur pārbaudīju sistēmu: Windows 10 vai jaunāka, macOS 11 (Big Sur) vai jaunāka, kā arī modernas Linux distribūcijas (piemēram, Ubuntu 20.04+).

Pārlūki: Google Chrome 89+, Mozilla Firefox 86+, Microsoft Edge 89+ un Safari 14+ (macOS). JavaScript pārlūkā jābūt ieslēgtam – viss kliēnta puses kods balstās uz tā, jo lietotne ir veidota kā vienas lapas SPA risinājums.

Aparatūra lokālai izstrādes videi: divkodolu procesors no 1 GHz, vismaz 2 GB RAM, 500 MB brīvas vietas diskā un platjoslas internets (no 5 Mbit/s). Lietotājam, kurš lietotni atver tikai pārlūkā, pietiek ar vēl mazākiem resursiem.

6.2. Sistēmas instalācija un palaišana

Lai sistēmu palaistu lokāli, datorā vajadzēs trīs lietas: Node.js (no 18.0), npm (no 9.0) un Git.

Tālāk secība ir vienkārša. Vispirms ar git clone nokopē projekta repozitoriju no GitHub. Tad atver projekta mapi un palaiž npm install, kas saliek visu nepieciešamo. Pēc tam datu bāzi inicializē ar divām komandām: node setup-db.cjs un node seed-db.cjs. Visbeidzot atver divus termināļa logus – vienā startē serveri ar node backend/server.cjs, otrā klientu ar npm run dev.

Pirmajā reizē, kad viss strādā, tīmekļa lietotne paveras adresē http://localhost:5173, bet REST API – http://localhost:3000. Administratora pieteikšanās dati ir norādīti README.md failā.

      return { success: true }

      return { success: false, error: error.response?.data?.error || 'Login failed' }
    }

      // Pēc veiksmīgas reģistrācijas lietotājs tiek novirzīts uz pieteikšanās lapu
      return { success: true, message: response.data.message }

      return { success: false, error: error.response?.data?.error || 'Registration failed' }
    }
  }

      }

  }

  return { user, token, isAuthenticated, login, register, googleLogin, logout, initializeAuth }

// Pārbauda, vai JWT tokenam tuvojas derīguma termiņa beigas (mazāk par 24 stundām)
function isTokenExpiringSoon(token) {
  try {
    // JWT payload ir Base64 kodēta tokena vidējā daļa
    const payload = JSON.parse(atob(token.split('.')[1]))
    // exp ir Unix laiks sekundēs — pārvēršam uz milisekundēm
    const expiresAt = payload.exp * 1000
    const oneDayMs = 24 * 60 * 60 * 1000
    return expiresAt - Date.now() < oneDayMs
  } catch {
    // Bojāts tokens — uzskata, ka termiņš nav drīz
    return false
  }
}

// Mainīgais glabā aktīvo atjaunošanas pieprasījumu, lai paralēli izsaukumi neradītu vairākus refresh

  let token = localStorage.getItem('token')
  if (token) {
    // Ja tokenam drīz beidzas derīgums un šis nav pats refresh pieprasījums — atjauno tokenu
    if (isTokenExpiringSoon(token) && !config.url.includes('/auth/refresh')) {
      if (!refreshing) {
        // Pieprasa jaunu tokenu, izmantojot pašreizējo (vēl derīgo) tokenu
        refreshing = axios.post(
          (import.meta.env.VITE_API_URL || 'http://localhost:3000') + '/api/auth/refresh',
          {},
          { headers: { Authorization: `Bearer ${token}` } }
        ).then(res => {
          // Saglabā jauno tokenu localStorage
          localStorage.setItem('token', res.data.token)
          return res.data.token
        }).catch(() => null).finally(() => { refreshing = null })
      }
      // Sagaida atjaunošanas pieprasījuma rezultātu
      const newToken = await refreshing
      if (newToken) token = newToken
    }
    // Pievieno Authorization galveni ar Bearer shēmu
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

// Atmiņā izveidots kratājs katras IP+ceļa kombinācijas pieprasījumu laika zīmogiem
const rateLimitStore = new Map();
// Augstākas kārtas funkcija — atgriež middleware ar norādīto pieprasījumu skaita ierobežojumu
const rateLimit = (maxRequests, windowMs) => (req, res, next) => {
  // Atslēga = klienta IP + pieprasītais ceļš (separāti limiti katram galapunktam)
  const key = `${req.ip}:${req.path}`;
  const now = Date.now();
  let record = rateLimitStore.get(key) || [];
  // Atstāj tikai tās laika zīmes, kas iekļaujas pašreizējā logā
  record = record.filter(t => t > now - windowMs);
  if (record.length >= maxRequests) {
    // Limits sasniegts — atgriež HTTP 429
    return res.status(429).json({ error: 'Too many requests. Please try again later.' });
  }
  // Reģistrē pašreizējo pieprasījumu un ļauj turpināt
  record.push(now);
  rateLimitStore.set(key, record);
  next();
};

app.post('/api/auth/register', rateLimit(5, 15 * 60 * 1000), async (req, res) => {
  const { name, email, password, phone } = req.body;
  // Pamata pārbaudes — obligātie lauki
  if (!name || !email || !password)
    return res.status(400).json({ error: 'Please fill in all required fields.' });
  if (name.trim().length < 2)
    return res.status(400).json({ error: 'Name must be at least 2 characters long' });
  // E-pasta formāta validācija ar regulāro izteiksmi
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  if (!emailRegex.test(email))
    return res.status(400).json({ error: 'Please enter a valid email address' });
  // Paroles stipruma pārbaude
  const passwordError = validatePassword(password);
  if (passwordError) return res.status(400).json({ error: passwordError });

  // Pārbauda, vai e-pasts jau eksistē datu bāzē (UNIQUE ierobežojuma sagaidāšana)
  db.get('SELECT id FROM users WHERE email = ?', [email], async (err, row) => {
    if (err) return res.status(500).json({ error: 'Database error' });
    if (row) return res.status(400).json({ error: 'Email already registered' });
    // Jauc paroli ar bcrypt (10 salt rounds = standarta drošības līmenis)
    const hashedPassword = await bcrypt.hash(password, 10);
    // Ievieto jauno lietotāju datu bāzē

      'INSERT INTO users (name, email, password, phone) VALUES (?, ?, ?, ?)',
      [name.trim(), email.toLowerCase(), hashedPassword, phone || null],
      function(err) {
        if (err) return res.status(500).json({ error: 'Failed to create account' });
        // Atgriež 201 Created statusu
        res.status(201).json({ message: 'Account created successfully' });

    if (!user) return res.status(401).json({ error: 'Invalid email or password' });

    const validPassword = await bcrypt.compare(password, user.password);

    const token = jwt.sign(
      { id: user.id, email: user.email, name: user.name, role: user.role },
      JWT_SECRET,
      { expiresIn: '7d' }
    );
    // Atgriež tokenu un lietotāja pamatdatus klientam
    res.json({ token, user: { id: user.id, email: user.email, name: user.name, role: user.role } });
  });

      if (user) {
        // Esošs lietotājs — ja konts izveidots ar e-pastu, pievieno Google ID
        if (!user.google_id) {
          db.run('UPDATE users SET google_id = ? WHERE id = ?', [googleId, user.id]);
        }
        return issueToken(res, user, false);
      }
      // Jauns lietotājs — izveido kontu bez paroles (Google saglabā autentifikāciju)
      db.run(
